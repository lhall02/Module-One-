Artifact Description

Why This Artifact Was Selected

The selected artifact is the trip data portion of the Travlr application. A travel application depends on searching, organizing, filtering, grouping, and retrieving data efficiently, so the API controller was a strong place to demonstrate algorithms and data structures in a practical project.

The original API primarily returned trip records. The enhanced version adds several features that make the data layer more useful and show how algorithmic choices affect behavior, performance, and maintainability.

Enhancement

Algorithms and Data Structures Added

Area Before After
Search Basic retrieval only. Binary search by trip code after sorting records by code.
Filtering No advanced query filtering. Multi-criteria filters for resort, price range, and start-date boundaries.
Grouping No summary data. Resort grouping with count, average price, minimum price, and maximum price.
Performance Repeated requests always queried the database. Map-based TTL cache reduces repeated work for common trip-list requests.
Routes Limited API surface. Additional endpoints for code lookup, filtering, grouping, date ranges, and cache clearing.

Reflection

Learning and Trade-Offs

Through this enhancement, I learned that algorithms and data structures are useful beyond classroom examples. Binary search, caching, filtering, grouping, and date-range queries all solve realistic problems in the application. I also learned that each design choice has trade-offs: caching can improve speed but introduces stale-data risk, while binary search is efficient but requires sorted input.

The enhancement supports the algorithms and data structures outcome because it uses searching, grouping, caching, and range logic to solve data access problems. It also supports the tools and techniques outcome through RESTful API design and MongoDB aggregation, and it supports the security outcome through input validation, error handling, and safer query construction.
